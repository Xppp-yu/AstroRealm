### 第4天：2026年3月16日
**目标**：设计安全架构

**关键决策**：**使用Docker沙盒**

**为什么选择Docker**：
1. 环境隔离（Agent搞不坏我的系统）
2. 资源限制（防止Agent占用太多资源）
3. 易于清理（用完就删，不留痕迹）

**学习Docker**：
- Docker基础概念：镜像、容器、卷
- Dockerfile编写
- docker-compose使用

**第一个Dockerfile**：
```dockerfile
FROM ubuntu:22.04

# 安装基础工具
RUN apt-get update && apt-get install -y \
    python3 \
    python3-pip \
    git \
    vim

# 创建非root用户（安全考虑）
RUN useradd -m -s /bin/bash astro
USER astro
WORKDIR /home/astro

# 安装Python依赖
COPY requirements.txt .
RUN pip3 install -r requirements.txt
```

**遇到的问题**：
- Docker镜像下载慢（换国内镜像）
- 权限配置复杂（用户切换）
- 文件挂载问题

**收获**：
- 理解了容器化技术的价值
- 学会了基本的Docker操作
- 建立了安全意识

### 第5天：2026年3月17日
**目标**：设计系统架构

**核心问题**：如何平衡本地和云端？

**我的需求分析**：
1. **数据安全**：核心数据必须在本地
2. **随时可用**：手机能随时访问
3. **计算能力**：复杂任务需要云端计算
4. **成本考虑**：学生预算有限

**架构草图**：
```
本地（安全区）：
  - 核心数据存储
  - 隐私信息
  - 本地轻量任务

云端（计算区）：
  - 临时计算环境
  - 7×24小时在线
  - 复杂任务处理
```

**技术选型思考**：
- **通信协议**：HTTP vs WebSocket
  - 选择HTTP：简单，容易调试
- **数据同步**：实时 vs 定期
  - 选择定期同步：降低复杂度
- **存储方案**：数据库 vs 文件系统
  - 选择文件系统：简单直观

**设计原则确立**：
1. **本地主权原则**：所有数据最终归属本地
2. **云端临时原则**：云端只做计算，不留数据
3. **简单优先原则**：能用简单方案就不用复杂方案

# local/controller.py 简化版
class LocalController:
    def __init__(self):
        self.tasks = []
        self.results = {}
        
    def receive_task(self, task):
        """接收任务"""
        # 安全检查
        if self._is_sensitive(task):
            return self._process_locally(task)
        else:
            return self._send_to_cloud(task)
    
    def _is_sensitive(self, task):
        """判断是否为敏感任务"""
        sensitive_keywords = ['密码', '密钥', '隐私']
        return any(kw in task for kw in sensitive_keywords)
```

class StudentAwareScheduler:
    """考虑学生作息的任务调度器"""
    
    def schedule_task(self, task):
        # 获取当前时间
        current_hour = datetime.now().hour
        
        # 上课时间（8:00-12:00, 14:00-18:00）
        if 8 <= current_hour < 12 or 14 <= current_hour < 18:
            # 上课时间，只处理紧急任务
            if task['priority'] == 'urgent':
                return 'process_now'
            else:
                return 'delay_until_evening'
        
        # 晚上学习时间（19:00-23:00）
        elif 19 <= current_hour < 23:
            # 学习时间，优先处理学习相关任务
            if task['type'] == 'study':
                return 'process_now'
            else:
                return 'process_normal'
        
        # 休息时间（23:00-8:00）
        else:
            # 休息时间，只处理非常紧急的任务
            if task['priority'] == 'critical':
                return 'process_now'
            else:
                return 'delay_until_morning'
```

## 🧠 思维成长记录

### 技术思维的变化

**第1周**：点状思维
- 只关注具体问题的解决
- "这个命令怎么用？"
- "这个错误怎么修？"

**第2周**：线状思维  
- 开始关注流程和连接
- "这个功能怎么实现？"
- "模块之间怎么通信？"

**第3周**：面状思维
- 考虑整体架构和设计
- "系统应该怎么组织？"
- "如何保证扩展性和维护性？"

### 学习方法的改进

**最初的错误方法**：
- 遇到问题就百度，复制粘贴解决方案
- 不深入理解原理
- 不做笔记，下次遇到同样问题还要重新查

**改进后的方法**：
1. **先理解问题**：弄清楚问题的本质
2. **多种方案比较**：不止找一个解决方案
3. **理解原理**：不只是会用，还要知道为什么
4. **做好记录**：把解决方案详细记下来
5. **定期复习**：每周回顾学到的知识

### 项目管理的学习

**教训1**：不要一开始就追求完美
- 第一个版本很烂，但至少能跑
- 先实现核心功能，再优化细节
- 快速迭代比完美设计更重要

**教训2**：文档和代码一样重要
- 不写文档，一个月后自己都看不懂
- 好文档能节省大量调试时间
- 文档要随着代码一起更新

**教训3**：测试不是可选的
- 手动测试效率低，容易遗漏
- 自动化测试能保证质量
- 测试代码也是文档的一部分

**表现**：看到技术文章，80%的术语都不懂

**应对策略**：
1. **建立知识地图**：画出需要学习的知识点
2. **分层学习**：先学基础，再学高级
3. **实践驱动**：边做项目边学习
4. **不求甚解**：先会用，再深入理解

### 困难2：时间管理困难
**表现**：专业课、项目、自学时间冲突

**应对策略**：
1. **时间块分配**：固定时间段做固定事情
2. **优先级排序**：重要且紧急的事情先做
3. **利用碎片时间**：课间、排队时学习
4. **合理放弃**：不是所有事情都要做到完美

**表现**：遇到问题不知道问谁

**应对策略**：

2. **阅读官方文档**：最权威的信息源
3. **分析开源项目**：学习别人的实现
4. **记录问题**：建立自己的知识库

**表现**：遇到困难想放弃

**应对策略**：
1. **小目标激励**：完成一个小功能就奖励自己
